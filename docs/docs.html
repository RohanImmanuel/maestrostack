Overview

MaestroStack is a config-driven CLI that wraps BrowserStack App Automate's Maestro workflow. Instead of uploading the app, zipping and uploading the suite, copying URLs and hand-building a payload by hand each time, you describe the run once and trigger it with a single command.

terminal
maestrostack run

It runs the same way on your machine and in CI, on macOS, Linux and Windows.

Install

MaestroStack is distributed on npm. It needs Node.js 18 or newer.

terminal
npm install -g maestrostack

You can also run it without a global install using npx maestrostack <command>.

Quick start

  1. Scaffold a config with maestrostack init --android.
  2. Provide BrowserStack credentials as environment variables.
  3. Validate locally, then preview with a dry run.
  4. Trigger the real build.
terminal
maestrostack init --android

export BROWSERSTACK_USERNAME=...
export BROWSERSTACK_ACCESS_KEY=...

maestrostack validate
maestrostack run --dry-run
maestrostack run

Project layout

MaestroStack encourages a simple structure. Maestro flows live in subfolders; the tool config sits at the root.

mobile-tests/
mobile-tests/
  maestrostack.yml      # tool config, not a Maestro flow
  smoke/
    login.yml           # Maestro flow
    signup.yml
  regression/
    checkout.yml
  apps/
    app-release.apk
Root YAML rule. Root-level YAML files are treated as config, not Maestro flows. Keep flows in subfolders so maestrostack.yml never gets confused with smoke/login.yml.

Config file

maestrostack init writes a starter maestrostack.yml. Here is a full example with every block.

maestrostack.yml
version: 1

auth:
  username: ${BROWSERSTACK_USERNAME}
  accessKey: ${BROWSERSTACK_ACCESS_KEY}

platform: android            # android | ios

app:
  source: upload             # upload | app_url
  path: ./apps/app-release.apk
  customId: SampleApp

suite:
  root: .
  packageName: Flows.zip
  customId: SampleTest
  include:
    - smoke/**/*.yml
    - regression/**/*.yml
  exclude:
    - apps/**

run:
  project: Maestro_Test
  devices:
    - Samsung Galaxy S20-10.0
    - Google Pixel 7-13.0
  executeMode: explicit      # explicit | main
  maxParallel: 2             # optional: max devices at once
  execute:
    - smoke/login.yml
    - regression/checkout.yml
  options:
    networkLogs: true
    deviceLogs: true

Use a different file with the global -c flag, for example maestrostack run -c maestrostack.staging.yml.

App source

The app.source field selects how MaestroStack resolves the app under test.

sourceRequiredBehaviour
uploadpathUploads a local .apk (Android) or .ipa (iOS).
app_urlappUrlReuses an existing bs:// app reference.
custom_idcustomIdReferences the app by custom id; BrowserStack resolves it to the latest upload with that id. No upload.

For upload, validation checks that an Android run points at a .apk and an iOS run at an .ipa, and that the file exists before anything is uploaded. app_url and custom_id reference an app already on BrowserStack, so nothing is uploaded for them.

maestrostack.yml
app:
  source: custom_id
  customId: SampleApp   # latest app uploaded with this id

Execute modes

Explicit mode (recommended)

List the flow files in run.execute. MaestroStack sends them as the BrowserStack execute array. No root main.yaml is needed.

maestrostack.yml
run:
  executeMode: explicit
  execute:
    - smoke/login.yml
    - regression/checkout.yml

Main mode

Omit run.execute and provide a main.yaml at the suite root. BrowserStack runs it as the entrypoint. Validation fails early if main.yaml is missing.

maestrostack.yml
run:
  executeMode: main

Parallel execution

Set run.maxParallel to cap how many of your listed devices run at once. MaestroStack splits run.devices into sequential batches of at most that many devices, submits one build per batch, and waits for each build to finish before starting the next.

maestrostack.yml
run:
  maxParallel: 2
  devices:
    - Samsung Galaxy S20-10.0
    - Google Pixel 7-13.0
    - iPhone 15-17.0
    - OnePlus 9-11.0

With four devices and maxParallel: 2, that runs two builds of two devices each, one after the other.

  • Every device runs the full suite; nothing is assigned randomly.
  • BrowserStack has no native per-build concurrency cap, so MaestroStack enforces it with sequential batched builds (polling each build's status until it completes).
  • How many sessions can actually run at once depends on your BrowserStack plan's parallel limit; maxParallel caps concurrency at or below that.
  • A batched run is long-running: it blocks until every batch finishes. Without maxParallel, run submits a single build and returns immediately.
  • Override per run with --max-parallel <n>.
Account parallels still apply. Within a batch, BrowserStack runs sessions up to your plan's purchased parallels; if a batch is larger than that, the extra sessions queue until a slot frees up.

Devices

List one or more real devices in run.devices. A single build request fans out across all of them. At least one device is required.

maestrostack.yml
run:
  devices:
    - Samsung Galaxy S20-10.0
    - Google Pixel 7-13.0

For iOS, list iOS device strings such as iPhone 15-17.0. MaestroStack does not validate names against BrowserStack's catalogue, so use the device strings from your BrowserStack dashboard.

Authentication

Credentials are referenced as ${VAR} tokens in the config and resolved from the environment. A .env file in the working directory is loaded automatically.

.env
BROWSERSTACK_USERNAME=my_username
BROWSERSTACK_ACCESS_KEY=my_access_key
Fail fast, stay quiet. If a referenced variable is missing, MaestroStack stops before any upload. Secrets are never printed, including in the dry-run payload preview.

Commands

CommandDescription
maestrostack initCreate a starter config. Flags: --android, --ios, --force.
maestrostack validateValidate config, suite structure, app and devices.
maestrostack packageDiscover flows and build the suite zip without uploading.
maestrostack upload-appUpload only the app and print its app_url.
maestrostack upload-suitePackage and upload only the suite and print its test_suite_url.
maestrostack runPackage, upload and trigger a build.

Global options

  • -c, --config <path> uses a specific config file.
  • --debug turns on verbose logging.

The run command

maestrostack run is the main command. It supports overrides and a safe preview mode.

OptionDescription
--dry-runValidate, show the files that would be zipped and print the build payload. No API calls.
--device <name>Override run.devices. Repeatable.
--execute <path>Override run.execute. Repeatable. Forces explicit mode.
--max-parallel <n>Override run.maxParallel. Run at most N listed devices at once.
terminal
maestrostack run -c maestrostack.android.smoke.yml
maestrostack run --device "Google Pixel 7-13.0"
maestrostack run --execute smoke/login.yml --dry-run

How a run works

  1. Loads .env and the config, substitutes ${VAR} tokens, and validates everything.
  2. Discovers flow files, ignoring root-level YAML, node_modules and .git.
  3. Zips the suite under a Flows/ prefix, preserving folder structure with forward-slash paths.
  4. Resolves the app by uploading the local binary or reusing an existing bs:// URL.
  5. Uploads the suite zip.
  6. Posts the build to /app-automate/maestro/v2/{android|ios}/build and prints the build ID and its BrowserStack dashboard URL.

Development

Clone the repo and use the standard scripts. The BrowserStack integration is covered by tests with mocked HTTP, so the full package-to-payload path runs without a live account.

terminal
npm install
npm run build       # bundle with tsup
npm run typecheck   # tsc --noEmit
npm test            # vitest

FAQ

What is MaestroStack?

MaestroStack is a config-driven command-line tool that packages your Maestro flow folders, uploads the app and test suite to BrowserStack App Automate, and triggers Maestro builds across one or more real devices from a single command.

How do I run Maestro tests on BrowserStack?

Install MaestroStack with npm install -g maestrostack, create a config with maestrostack init, set your BROWSERSTACK_USERNAME and BROWSERSTACK_ACCESS_KEY, then run maestrostack run. It uploads the app and suite and starts the build for you.

Does MaestroStack work on Windows and Linux?

Yes. It runs the same on macOS, Linux and Windows and needs Node.js 18 or newer. All paths are normalized internally and there are no platform-specific shell calls.

Do I need a main.yaml file?

Not in explicit mode, which is the default - you list the flows in run.execute. A main.yaml at the suite root is only required when you set executeMode: main.

How does parallel execution work?

Set run.maxParallel to cap how many of your listed devices run at once. MaestroStack splits the devices into sequential batches of that size and waits for each build to finish before starting the next, so concurrency stays controlled and every device runs the full suite.

Is MaestroStack free and open source?

Yes. It is open source under the MIT license and published on npm as maestrostack.

What do I need to use it?

A BrowserStack App Automate account with API credentials, a Maestro test suite (flow files in subfolders), and either a local app binary (.apk or .ipa) or an existing bs:// app URL.
